## 上半段：大白话版

## 1. 这个项目到底是干什么的

这个项目说白了，就是一个“**帮人看设备、记设备、发现问题、提醒问题**”的小型网络管理系统。

平时在一个机房、实验室、公司网络里，可能会有很多设备，比如：

- 路由器
- 交换机
- 网关
- 服务器
- 其他有 IP 地址、能联网的设备

如果只靠人去一个一个记，谁在线、谁掉线、谁 CPU 高了、谁内存高了、谁最近出过问题，其实很麻烦。  
所以这个系统的作用，就是把这些事情集中到一个软件里统一管理。

你可以把它理解成一个“**网络设备管家**”：

- 它能记住设备是谁
- 它能定时看设备还活着没有
- 它能记录设备的运行情况
- 它能在设备出问题的时候发出告警
- 它还能把设备配置备份下来，方便查看和对比

这个项目是一个毕设项目，所以目标不是做成那种商用级、特别庞大、特别复杂的网管平台，而是做成一个：

- 功能完整
- 思路清楚

- 自己讲得明白

的成品。

换句话说，这个项目追求的是：

**不要太深太重，但也不能像玩具。**

所以我给它定的方向是：

- 基础功能要完整
- 核心流程要打通
- 有一定真实设备管理味道
- 细节上不能太粗糙

## 3. 这个系统可以帮人做什么

用最直白的话讲，它主要做 5 件事。

### 第一件：管理设备

你可以把设备录进去，比如：

- 设备 IP
- 设备类型
- SNMP community
- SSH 用户名和密码

录进去以后，系统就知道“我要管哪些设备”。

### 第二件：看设备是不是在线

系统会定时去看设备能不能连通。  
如果设备断了、网络不通了、Ping 不通了，系统就会把它记下来，并且发出离线告警。

### 第三件：看设备运行状态

除了看它通不通，系统还会尽量去读：

- CPU
- 内存
- 接口状态

这样就不只是“知道设备死没死”，而是还能知道“设备现在忙不忙、压力大不大”。

### 第四件：出问题时提醒人

比如：

- 设备离线了
- CPU 太高了
- 内存太高了

系统会自动生成告警。  
而且现在不是那种一有问题就疯狂刷屏的低级做法，而是会做去重，也会做恢复。

意思就是：

- 同一个问题短时间内不会一直重复报警
- 问题好了以后，系统会把它标记成“已恢复”

### 第五件：备份设备配置

系统还可以去尝试通过 SSH 连设备，把配置抓回来，保存到数据库里。

这样做的好处是：

- 以后想查配置，可以直接看
- 想比较两次配置哪里不一样，也能对比
- 做演示时更像一个完整系统，不只是监控页面

## 4. 这个系统是怎么组成的

这个项目不是网页，也不是手机 App，它是一个 **C/S 架构** 的桌面系统。

你可以把它分成 3 个部分来理解：

### 1. 客户端

客户端就是你看到的界面，也就是老师演示时真正看到的那个软件窗口。

它负责：

- 显示页面
- 点按钮
- 查数据
- 看图表
- 看告警
- 管设备

### 2. 服务端

服务端可以理解成“后台大脑”。

它负责：

- 接收客户端发来的请求
- 跟数据库交互
- 定时采集设备状态
- 生成告警
- 做配置备份

### 3. 数据库

数据库可以理解成“项目的记事本”。

它负责把这些东西存起来：

- 设备信息
- 监控日志
- 告警记录
- 配置备份内容

所以整个过程其实就是：

**客户端下命令 -> 服务端处理 -> 数据库存结果 -> 客户端再显示出来。**

你可以按设备、按日期范围去查，看看设备在某一段时间里：

- 是否在线

这里要注意：

## 7. 这个项目完整跑起来时，实际是在干什么

如果从老师打开系统开始，一整套流程可以这样理解：

### 第一步：先连接服务端

客户端打开后，先连到服务端。

### 第二步：服务端从数据库里拿设备列表

服务端知道自己现在要管理哪些设备。

### 第三步：定时采集设备状态

系统会定时去做两层检查：

- 先看设备通不通
- 通了以后再继续读 CPU、内存、接口等更详细的数据

### 第四步：把结果存下来

每次采集到的数据都会写进数据库，形成历史记录。

### 第五步：如果发现异常，就生成告警

比如：

- 设备离线
- CPU 太高
- 内存太高

这些都能自动生成告警。

### 第六步：如果后面恢复了，再标记成已恢复

这说明系统不是只会“看到坏消息”，它还能“知道问题已经好了”。

### 第七步：前端把这些内容展示出来

最后老师看到的，就是：

- 首页统计
- 设备状态
- 告警中心
- 监控日志
- 配置备份与差异对比

## 下半段：专业版

## 1. 项目定位

本项目是一个基于 Java 17 的 C/S 架构网络设备管理系统，面向毕业设计场景，目标是在可控难度范围内实现：

- 设备管理
- 可达性监测
- SNMP 状态采集
- 告警生成与生命周期管理
- 配置备份与差异比对
- 客户端可视化展示

项目强调“**功能链路完整、实现不过度堆砌、具备答辩可讲性**”。

---

## 2. 总体架构

系统采用典型的三层思路：

### 1. 客户端层

- 技术：JavaFX
- 作用：承载设备管理、告警中心、监控日志、配置备份、系统概览等交互页面

### 2. 服务端层

- 技术：Java 17 + Socket + JSON
- 作用：提供业务处理、数据库访问、定时采集、告警生成、配置备份等能力

### 3. 数据持久层

- 技术：MySQL 8+ + HikariCP
- 作用：存储设备信息、监控记录、告警记录、配置备份内容

项目模块划分如下：

- `common`：协议对象、JSON 工具、通用模型
- `server`：服务端主逻辑、DAO、Socket 处理、监控调度
- `client`：桌面客户端 UI 与交互逻辑

---

## 3. 核心业务流程

### 1. 设备录入

用户在客户端录入设备的：

- IP
- 类型
- SNMP community
- SSH 用户名/密码

这些数据会由服务端写入 `device` 表。

### 2. 周期采集

服务端的监控调度器会定期扫描设备列表，并对每台设备独立提交采集任务。

采集流程为：

1. 先通过 `PingChecker` 判断设备可达性
2. 若在线，则继续调用 `SnmpCollector`
3. 读取 CPU、内存、接口状态
4. 将结果写入 `monitor_log`
5. 根据阈值判断是否触发告警

### 3. 告警处理

当设备离线或监控指标超阈值时，系统生成告警。  
同类告警在短时间窗口内会做去重；若状态恢复，则自动标记为已恢复。

### 4. 配置管理

配置管理模块会尝试通过 SSH 连接目标设备，执行配置备份命令并保存文本内容；客户端支持对备份结果进行查询、差异对比与演示级恢复。

## 7. 本项目的重点增强点

### 1. SNMP 监控从“模拟”走向“真实采集”

本项目引入 `SNMP4J` 实现设备状态采集，并采用可配置 OID 的方式兼顾通用性与可控复杂度。  
实现策略不是做厂商深度适配，而是优先支持标准接口 OID，并对 CPU / 内存走配置化方案。

### 2. 监控调度改为并行执行

为了避免一台设备阻塞整轮采集，调度器采用工作线程池按设备维度并行执行采集任务，并设置单轮最大时长约束。

### 3. 告警具备完整生命周期

告警不再停留在“生成一条记录”这个层面，而是具备：

- 去重
- 确认
- 自动恢复
- 状态展示

这使得系统更接近真实运维场景。

### 4. 配置备份能力增强

通过 `JSch` 实现 SSH 备份链路，配置内容真实入库；恢复功能采用演示模式，既满足流程完整性，又控制毕设风险。

### 5. UI 层完成成品化补强

包括但不限于：

- 首页总览
- 趋势图展示
- 拓扑简图
- 自动刷新
- 分页
- 导出
- 自定义弹窗
- 表格与按钮交互优化

## 8. 设计取舍与难度控制

### 已实现

- 完整的客户端 / 服务端 / 数据库闭环

这种取舍的好处是：

- 难度可控

## 10. 答辩时的专业讲法

如果老师希望你用更专业的方式介绍，可以按下面这一段说：

“本项目是一个基于 Java 17 的网络设备管理系统，采用 C/S 架构，客户端基于 JavaFX 实现，服务端通过 Socket 提供 JSON 请求响应接口，底层使用 MySQL 完成数据持久化。  
系统围绕设备管理、状态采集、告警处理和配置管理四条主线展开。  
在监控方面，服务端通过定时调度器对设备进行并行采集，先执行可达性检测，再结合 SNMP4J 获取 CPU、内存和接口状态，并将采集结果写入 `monitor_log`。  
在告警方面，系统支持离线、CPU 过高、内存过高等告警的生成、去重与自动恢复。  
在配置管理方面，系统通过 JSch 建立 SSH 会话执行备份命令，并支持配置差异比对与演示级恢复。  
整体设计上我重点追求功能闭环完整、实现难度可控、演示效果清晰，适合作为毕业设计展示。”
