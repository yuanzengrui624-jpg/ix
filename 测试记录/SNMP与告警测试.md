# SNMP 与告警测试记录

> 日期：2026-03-21
> 版本：V3 二阶段增强版

## 1. 本轮测试目标

这轮测试主要验证 5 件事：

1. 服务端是否已经从“串行采集”改成“并行采集”
2. SNMP 不通时，系统会不会卡死，还是会在 10 秒内正常结束
3. 监控结果是否能正确写入 `monitor_log`
4. 配置备份是否已经改成“真实 SSH 备份”，失败时是否明确报错
5. 配置恢复接口是否已经打通，并明确标识为“演示恢复”

---

## 3. 本地已实测结果

### 用例 1：SNMP 不通时的采集耗时

**目的：**
验证监控线程池改造后，设备即使 SNMP 不通，也能在 10 秒内结束一轮采集，不会把整个监控卡死。

**实测方法：**

1. 启动新版服务端
2. 等待监控调度器自动跑一轮
3. 读取服务端日志

**日志结果：**

```text
16:05:20.119 INFO  [monitor-worker] ... 设备 127.0.0.1 Ping 正常，但 SNMP 指标未采集到: SNMP 无响应或 community 不正确
16:05:20.120 INFO  [monitor-scheduler] ... 本轮监控完成: 设备 1 台, 用时 4030 ms
```

**结论：**

- 结果：通过
- 耗时：`4030 ms`
- 是否满足“≤10s”：满足
- 说明：
  - 设备在线，但 SNMP 没有响应
  - 系统没有卡死
  - 系统没有把整轮采集拖到 10 秒以上
  - 说明“先探活、再采集”的 SNMP 降级策略已经生效

---

### 用例 2：监控日志是否正确写入

**目的：**
验证采集结束后，`monitor_log` 是否能正确落库。

**实测 SQL：**

```sql
SELECT id, device_id, ping_status, cpu, mem, interface_status, collect_time
FROM monitor_log
ORDER BY id DESC
LIMIT 1;
```

**实测结果：**

```text
id=482
device_id=1
ping_status=1
cpu=NULL
mem=NULL
interface_status=NULL
collect_time=2026-03-21 16:05:16
```

**结论：**

- 结果：通过
- 说明：
  - `Ping` 正常，所以 `ping_status=1`
  - SNMP 未响应，所以 `cpu/mem/interface_status` 都是 `NULL`
  - 这符合当前设计：**能 Ping 通但 SNMP 不通时，系统只降级指标采集，不把设备直接判死**

---

### 用例 3：真实 SSH 备份失败时是否明确报错

**目的：**
验证配置备份已经改为“真实 SSH 备份”，并且在设备未配置 SSH 凭据时返回明确错误，而不是假装成功。

**调用接口：**

- Action：`CONFIG_BACKUP_DEVICE`
- 参数：`deviceId=1`

**实测结果：**

```json
{"id":"config_backup_device","ok":false,"data":null,"error":"设备未配置 SSH 用户名"}
```

**结论：**

- 结果：通过
- 说明：
  - 旧版是模拟生成配置文本
  - 新版会先检查 SSH 凭据，再尝试真实备份
  - 如果前提不满足，会明确失败，不再伪装成“备份成功”

---

### 用例 4：演示级恢复接口是否打通

**目的：**
验证配置恢复接口已经存在，并且当前版本明确是“演示恢复”，不会真正下发配置。

**调用接口：**

- Action：`CONFIG_BACKUP_RESTORE`
- 参数：`id=5`

**实测结果：**

```json
{"id":"config_backup_restore","ok":true,"data":{"success":true,"demoMode":true,"deviceIp":"127.0.0.1","deviceId":1,"backupId":5,"lineCount":31,"message":"已生成恢复演示流程，当前版本不会真正下发到设备。"},"error":null}
```

**结论：**

- 结果：通过
- 说明：
  - 恢复接口已经打通
  - 会返回目标设备、备份编号、配置行数等信息
  - 当前模式明确为 `demoMode=true`
  - 这符合毕设“有流程、有界面、有接口，但不过度危险”的收敛目标

## 5. 答辩环境建议补测项

由于本地只有 1 台 `127.0.0.1` 测试设备，且当前没有真实 SNMP agent / SSH 设备，下面这些场景建议在答辩前用真实网络设备、eNSP/GNS3、或 SNMP 模拟器再补一轮：

### 建议补测 1：CPU 超阈值告警

**目标：**

- CPU 指标超过阈值时，生成 `CPU_HIGH`
- 同设备同类型在短时间内只生成 1 条（去重）

**验证点：**

- 首次超阈值会生成告警
- 连续多轮超阈值不会重复刷很多条

### 建议补测 2：CPU / 内存恢复逻辑

**目标：**

- 当 CPU / 内存恢复正常时，原告警自动标记为“已恢复”

**验证点：**

- 告警状态由“未确认”变成“已恢复”
- 告警中心显示正确
- 未确认告警统计会下降

### 建议补测 3：接口表真实采集

**目标：**

- 采集 `ifName/ifDescr`
- 采集 `ifOperStatus`
- 采集 `ifSpeed`

**验证点：**

- `monitor_log.interface_status` 中确实写入 JSON
- 客户端“接口状态”列能看到类似 `3 UP / 1 DOWN`

### 建议补测 4：真实 SSH 备份

**目标：**

- 配置了正确 `ssh_user/ssh_pwd` 后，执行 `show running-config`
- 备份内容真实写入 `config_backup.content`

**验证点：**

- 备份不再报错
- 配置内容不是模拟文本，而是真实设备输出

## 6. 毕设答辩时可以怎么讲

如果老师问“你这个做得多真实”，可以这样回答：

1. **监控层不是纯模拟了**：现在已经接入了 SNMP4J，接口状态走标准 IF-MIB，CPU/内存支持配置 OID。
2. **为了控制毕设难度，做了收敛**：不同厂商 CPU/内存 OID 不统一，所以采用“可配置 OID + 不通时优雅降级”的方案。
3. **告警不是只会报，不会消**：现在有去重，也有恢复逻辑，告警生命周期是完整的。
4. **配置管理做了安全收敛**：备份走真实 SSH，恢复只做演示流程，不做危险的真实回写。
5. **性能上做了控制**：监控改为线程池并行采集，单轮耗时已经实测控制在 10 秒以内。
