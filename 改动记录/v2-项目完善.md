## 上半段：大白话版（0 基础也能看懂）

### 一句话总结

这次不是只修了几个小 bug，而是把这个项目从“主要靠模拟、很多地方像半成品”的状态，继续往“更像真正网络设备管理系统”的方向推进了一大步：它现在开始尝试读真实设备状态、告警不再只会报不会消、配置可以走真实 SSH 备份、界面也更像成品，还补了测试记录。

### 需求1：优先实现 SNMP 监控

**原文需求：**

```text
1.优先实现SNMP监控（第二阶段核心）：
引入SNMP4J，定义CPU、内存、接口表的OID。
修改监控线程池，为每个设备分配独立任务，避免阻塞。
将采集到的数据写入monitor_log，并根据阈值生成告警。
测试响应时间，确保≤10s。
```

**翻译成大白话：**

这段话说白了，就是不想再让系统自己瞎编一堆 CPU、内存、接口状态的数据了，而是希望系统开始真正去设备身上“打听消息”。  
而且不能因为一台设备慢、卡、没反应，就把其他设备全拖住。读到的数据还要存起来，超标了要自动报警，最后还得测一下速度，证明它不是慢吞吞的摆设。

**这次实际上做成了什么：**

- 我把系统接上了真正去设备“读状态”的这条路，让它开始尝试读真实监控数据，而不是只靠系统自己编数字。
- 监控流程变成了“先试一下设备通不通，再去读更详细的状态”。
- 设备的接口状态优先按比较通用的标准方式去读，这样不会一上来就绑死某一个厂商。
- CPU 和内存这两项做成了“可以改配置”的方式。也就是说，不同设备如果不一样，可以换读法，而不是把代码写死。
- 监控不再是一台一台慢慢排队跑，而是改成了并行处理，一台设备卡住，不会把整轮监控都拖死。
- 读到的数据会继续存进数据库里，如果 CPU 或内存太高，还是会自动出告警。
- 本地实际测下来，一轮监控大概 4 秒左右，没有超过需求里说的 10 秒。
- 需要实话实说的是：我本机现在没有一台真的会回这种监控数据的网络设备，所以本地已经验证到的是“读不到详细数据时系统也不会死机、不会乱判离线”；如果接上真实设备，就能读到更完整的数据。

### 需求2：完善告警机制

**原文需求：**

```text
2.完善告警机制：
增加告警去重（相同设备+类型在短时间内只生成一条）。
增加告警恢复逻辑（当指标恢复正常时，将原告警标记为已恢复）。
```

**翻译成大白话：**

这段话的意思其实很简单：  
第一，别让同一种问题一直刷屏。  
第二，别让告警只会冒出来，不会自己消掉。  
比如一台设备 CPU 一直高，不能一分钟来一条、一分钟来一条，把页面刷满；它后来恢复正常了，也应该让系统自己知道“哦，这个问题已经好了”。

**这次实际上做成了什么：**

- 同一台设备、同一种告警，在短时间内不会重复刷很多条。
- 去重不只是设备离线，CPU 过高、内存过高也一起管了。
- 告警现在不只是“未确认”和“已确认”，还多了“已恢复”这个状态。
- 如果设备重新在线了，原来的离线告警会自动恢复。
- 如果 CPU 或内存恢复正常了，对应的过高告警也会自动恢复。
- 告警中心现在能直接看出一条告警到底是“还没处理”“人工确认了”还是“系统自动判定已经恢复了”。

**翻译成大白话：**

说白了，就是别再只让系统“装样子”生成一段假配置了，而是要让它真的去设备那边连一下、把配置抓回来、保存起来。  
保存完之后，还要能拿两份配置做对比，看哪里变了。  
恢复这件事也要有个流程，但考虑到这是毕设，不能把它做成一个一不小心就真把设备改坏的危险功能。

**这次实际上做成了什么：**

- 我把配置备份从“系统自己模拟一段文本”，改成了“优先真的去连设备抓配置”。
- 系统现在会尝试登录设备，再去执行备份命令。
- 备份出来的内容会继续存进数据库里，不是随便丢掉。
- 如果设备没有填 SSH 用户名或密码，它不会装作成功，而是会明确告诉你：备份失败，原因是什么。
- 恢复接口已经加上了，也能走完整个恢复流程。
- 但是恢复目前是“演示模式”，不会真的把配置强行写回设备。这么做是为了毕设既完整，又不把风险做得太高。
- 两份配置做差异对比这个功能前面就已经做过了，这次继续保留，还是能直接看左右两边哪里不一样。

### 额外增加的优化（大白话）

下面这些不是这次原文里硬性写死的要求，但为了让项目看起来不像半成品，我顺手一起补强了：

1. **打开软件先让你填连接地址**

2. **加了一个首页总览**  
   现在一进来就能先看到设备多少台、在线多少、离线多少、多少条告警没处理，还配了图表。

   表格内容居中了，分页有了，滚动条更明显了，空白行也处理了，整体看起来顺眼多了。

   点刷新会显示“刷新中...”，点采集会显示“采集中...”，其他按钮也补了悬停和按下的颜色变化，不会再按了跟没按一样。

5. **弹窗不再丑了**  
   以前是系统默认那种灰灰白白的弹窗，现在换成了自定义弹窗，位置、样式、拖动体验都更自然。

6. **导出和对比这些小功能都补齐了**  
   告警和监控日志都能导出 CSV，配置备份不仅能看，还能做差异对比。

7. **顺手修了不少容易出事的小坑**  
   比如空数据崩溃、批量备份中途断掉、日期选反、表格文字变黑、按钮颜色不恢复这些问题，我都一起修掉了。

### 使用说明

下面这一部分按原始需求 `1~5` 逐条整理。每条都会先放原文，再说明这次实际怎么实现、哪些地方做了收敛、哪些地方沿用了前一轮已经做好的能力。

### 需求1：优先实现 SNMP 监控（专业版）

**原文需求：**

```text
1.优先实现SNMP监控（第二阶段核心）：
引入SNMP4J，定义CPU、内存、接口表的OID。
修改监控线程池，为每个设备分配独立任务，避免阻塞。
将采集到的数据写入monitor_log，并根据阈值生成告警。
测试响应时间，确保≤10s。
```

**实现情况：**

- 已在 `server/pom.xml` 中引入 `snmp4j`
- 新增 `server/src/main/java/com/netmgmt/server/monitor/SnmpCollector.java`
- 新增 `server/src/main/java/com/netmgmt/server/monitor/SnmpSample.java`
- 重构 `server/src/main/java/com/netmgmt/server/monitor/MonitorScheduler.java`
- 扩展 `server/src/main/resources/application.properties`，加入 SNMP OID、超时、重试、线程池等配置项

**技术说明：**

- 监控链路改为：`PingChecker` 先做可达性判断，在线设备再进入 `SnmpCollector`
- `SnmpCollector` 采用“先探活，再采集”的方式：
  - 先请求 `sysDescr` 做轻量探测
  - 设备有响应后再继续读取 CPU、内存和接口表
- 接口状态优先使用标准 IF-MIB 相关 OID：
  - `interfaceNameOid`
  - `interfaceDescrOid`
  - `interfaceStatusOid`
  - `interfaceSpeedOid`
- CPU 和内存采用“配置项驱动”的收敛方案：
  - CPU OID 支持直接配置
  - 内存支持直接百分比 OID，也支持 `used/total`、`available/total` 的兜底换算
- `MonitorScheduler` 从原来的串行轮询，改成：
  - 单独的调度线程
  - 固定大小的工作线程池
  - 每台设备独立提交采集任务
  - 每轮采集总时长受 `monitor.maxRunSeconds` 约束
- 采集结果仍然写入 `monitor_log`，不额外增加复杂表结构

**收敛处理：**

- 没做厂商深度绑定版 SNMP
- 当前方案是“接口状态优先走标准 OID，CPU/内存走可配置 OID，不支持时返回 `null` 并优雅降级”
- 这样更符合毕设“不要太深，但不能粗糙”的目标
